# New Company Deployment Skill

## Quick Reference

This skill helps create a new company project with:
- **Track 1**: Framework (global_scripts symlink → shared/)
- **Track 2**: Scripts (update_scripts symlink → shared/)
- **Track 3**: Configuration (project-specific)
- **Track 4**: Communication Protocol (nsql symlink → shared/) - Human-AI Confirmation Protocol

## Execution Flow

### Step 1.5: Existing Rawdata Detection (IMPORTANT)

> **Why this step?** Many new company projects already have rawdata from previous work or legacy setups.
> The rawdata folder (`rawdata_{COMPANY}`) may exist in unexpected locations — inside a Dropbox conflict
> copy like `data 2/`, in the project root, or in a sibling directory. Detecting it early ensures rawdata
> lands in the standard location before we proceed.
>
> **Only rawdata matters.** Other legacy files (old DuckDB, old parameters, database_to_csv) are ETL
> artifacts that will be regenerated by the pipeline. They should be ignored, not moved.

**Scan for existing rawdata:**

```bash
# Check 1: rawdata inside legacy/conflict data folders
find {PROJECT_PATH} -maxdepth 4 -type d -name "rawdata_{COMPANY}" 2>/dev/null
# Common hits: "data 2/local_data/rawdata_{COMPANY}", "data/local_data/rawdata_{COMPANY}"

# Check 2: rawdata in sibling directory (common initial drop location)
ls -d {PROJECT_PATH}/../rawdata_{COMPANY} 2>/dev/null

# Check 3: rawdata directly in project root
ls -d {PROJECT_PATH}/rawdata_{COMPANY} 2>/dev/null
```

**If rawdata is found:**

Use `AskUserQuestion` to confirm, then display Finder instructions:

### Step 2.5: Company Parameters Setup (3 REQUIRED Files)

> **Why this step?** The app depends on 3 company-specific parameter files. Without them:
> - `df_product_line.csv` missing → App crashes on startup (UI dropdowns and filtering depend on it)
> - `chinese_labels.yaml` missing → UI shows MAMBA-specific product attributes (e.g., "油容量", "溫度控制") that are meaningless for the new company
> - `brand_aliases.yaml` missing → Brand standardization uses MAMBA's competitors (e.g., Clasin Home, UMBRA) instead of the new company's competitors
>
> **Deep merge mechanism (SO_P016):** The parameter loading system merges APP-level files on top of Global defaults.
> You only need to override the company-specific sections — other sections inherit from Global automatically.

#### Step 2.5a-0: Product Line Discovery Gate (MUST RUN BEFORE 2.5a)

> **Critical:** Do not define `product_line_id` from partial observations (e.g., first few tabs or a single sheet section).
> You must complete full discovery first, otherwise wrong IDs create empty categories in app filters and ETL outputs.

**Mandatory discovery workflow:**

1. Read **all tab names** in the coding sheet.
2. Read the full product-line source tabs (commonly `產品類別彙整表`, `Product Line`, `進度表`; use all that exist).
3. Extract candidate product lines from the **entire tab range** (no row-limit sampling).
4. De-duplicate and normalize names, then build a discovery table:
   - `source_tab`
   - `source_product_line_name`
   - `proposed_product_line_id`
   - `coverage_status` (`RESOLVED`, `AMBIGUOUS`, `NO_SOURCE_MATCH`)
5. Validate coverage against real source structures (profile tabs / competitor headers / raw files).  
   If unresolved lines exist, ask user before finalizing `df_product_line.csv`.

**Gate rule (hard stop):**
- If any line is `AMBIGUOUS` or `NO_SOURCE_MATCH`, do **not** finalize `df_product_line.csv` silently.
- Either:
  - resolve mapping with user confirmation, or
  - set `included=FALSE` with a clear note of why.

#### Step 2.5a: df_product_line.csv (REQUIRED — App startup dependency)

**Data source priority:**

1. **Coding Sheet URL** (from Step 1) → Use `WebFetch` to extract product categories, convert to CSV
2. **KEYS.xlsx** (if available in rawdata) → Extract product classification from it
3. **Manual input** → Use `AskUserQuestion` to ask for product line list

**Required format:**

```csv
product_line_name_english,product_line_name_chinese,product_line_id,included
all,所有產品,all,TRUE
{product_1_english},{product_1_chinese},{product_1_id},TRUE
...
```

**Rules:**
- First row MUST be `all,所有產品,all,TRUE`
- `product_line_id` decision MUST happen **after Step 2.5a-0 discovery gate**
- `product_line_id` should use 3-character lowercase abbreviation by default (e.g., `sgl`, `hlm`)
- If source uses explicit coded keys (e.g., `001_xxx`), keep a one-to-one mapping record and confirm final ID scheme with user before writing CSV
- Output: `{PROJECT_PATH}/data/app_data/parameters/scd_type1/df_product_line.csv`

### Step 4: Initialize Git and Create Symlinks

**Important**: If Step 1.5 found legacy data folders (e.g., `data 2/`) that the user chose not to move,
add them to `.gitignore` BEFORE the initial commit. Also add any Dropbox conflict copies.

**Architecture**: l4_enterprise uses a **shared/ symlink pattern** (not subrepo clones).
The tier-level `shared/` directory contains the actual repos, and each company project
creates symlinks pointing to them. This avoids duplicating ~63MB of code per company.

```
{TIER_PATH}/
├── shared/                    # Actual repos (managed at tier level)
│   ├── global_scripts/        # Track 1
│   ├── update_scripts/        # Track 2
│   └── nsql/                  # Track 4
├── {COMPANY}/
│   └── scripts/
│       ├── global_scripts -> ../../shared/global_scripts   # Symlink
│       ├── update_scripts -> ../../shared/update_scripts   # Symlink
│       └── nsql -> ../../shared/nsql                       # Symlink
└── {OTHER_COMPANY}/
    └── scripts/               # Same symlink pattern
```

```bash
cd {PROJECT_PATH}

# Initialize git at project ROOT (not in scripts/)
git init

# Create .gitignore FIRST (use gitignore_template + add any legacy folder exclusions)
# If "data 2/" or similar exists, add to .gitignore:
#   data 2/

# Initial commit with .gitignore + parameter files
git add .gitignore data/ logs/ docs/
git commit -m "Initial commit: directory structure and parameter files"

# Create GitHub repo via CLI (naming convention: ai_martech_{TIER}_{COMPANY})
gh repo create kiki830621/ai_martech_{TIER}_{COMPANY} --private --source=. --remote=origin

# Disable GitHub Issues (issues are tracked centrally in global_scripts repo)
gh repo edit kiki830621/ai_martech_{TIER}_{COMPANY} --enable-issues=false

# Create symlinks to shared/ (NOT subrepo clones)
mkdir -p scripts
ln -s ../../shared/global_scripts scripts/global_scripts
ln -s ../../shared/update_scripts scripts/update_scripts
ln -s ../../shared/nsql scripts/nsql

# Verify symlinks resolve correctly
ls scripts/global_scripts/00_principles/INDEX.md && echo "global_scripts OK"
ls scripts/update_scripts/Makefile && echo "update_scripts OK"
ls scripts/nsql/protocol.yaml && echo "nsql OK"
```

> **Why symlinks, not subrepos?** In Dropbox-synced environments, subrepo clones create
> ~63MB of duplicated files per company and cause sync conflicts. Symlinks to `shared/`
> ensure all companies use the same code with zero duplication.

> **Naming Convention**: `ai_martech_l4_ACME` for l4_enterprise tier, `ai_martech_l3_ACME` for l3_premium, etc.

> **NSQL**: The NSQL protocol provides a Human-AI Confirmation Protocol for unambiguous communication between users and AI systems. See `scripts/nsql/docs/` for documentation.
