# 會議逐字稿校正指南

## 校正流程

### Step 1：讀取原始 SRT

1. **讀取原始 .srt 逐字稿**
   - **保留所有原始時間碼** — 後續需要用來對應句子
   - 識別語音辨識的常見錯誤
   - 記錄所有出現的 Speaker 標籤

2. **讀取術語對照表**
   - 參見 `references/terminology.md`
   - 如有會議議程或相關文件，一併讀取作為校正依據

3. **驗證 AI 模型名稱**
   - AI 模型名稱更新快速，不要假設某個模型名是「辨識錯誤」
   - 例如：GPT-5 mini 是真實存在的模型，不是 GPT-4o-mini 的誤認
   - 若遇到不確定的模型名，使用 `/openai-docs-guide` 或 `/claude-docs-guide` 查詢最新模型清單
   - 也可參考 `l4_enterprise/shared/global_scripts/08_ai/` 中實際使用的模型名

### Step 2：校正內容

#### 2.1 說話者標籤標準化

Plaud 轉錄會產生 `Speaker 1`、`Speaker 2` 等匿名標籤，以及部分已辨識的名字。

| 處理方式 | 說明 |
|---------|------|
| 已辨識名字 | 保留（如 `che:`、`林郁翔:`） |
| 匿名 Speaker | 若能從上下文推斷身份，加註 `Speaker 1 (推測:XXX):` |
| 無法推斷 | 保留原始 `Speaker N:` |

#### 2.2 專有名詞校正

根據 `references/terminology.md` 校正以下類型的錯誤：

| 錯誤類型 | 範例 |
|---------|------|
| 公司名同音異字 | 漫八 → 曼巴 (MAMBA)、像創/相創 → 向創 (QEF) |
| 產品名 | 保留英文原名：BrandEdge、InsightForge、VitalSigns |
| 技術術語 | DuckDB、Posit Connect、ETL、pipeline |
| 英文縮寫 | API、GPT、RFM、NES |
| 人名 | 根據已知團隊成員校正 |

#### 2.3 去除贅詞（Stop Words）

移除以下口語贅詞，但保持語意完整：

```
嗯、啊、呃、欸
那個、這個、那種
就是說、也就是說（過度重複時）
然後呢、然後、接著（過度重複時）
對不對、對吧、是不是
好、好的（非確認用途時）
這樣子、就這樣
所以說、基本上、其實
你知道、你看
```

#### 2.4 改善語順

- 將口語的倒裝句改為書面語順序
- 適度添加標點符號
- **不要合併不同說話者的發言**
- 保持會議對話的自然節奏

### Step 3：重組為「每句一個時間戳」

#### 核心原則

- **以「一句話」為單位** — 一個完整的語意單位
- 每個字幕區塊最多 1-2 行，約 30-50 個中文字
- **時間碼必須來自原始 SRT** — 絕對不能憑空估算或虛構
- **說話者切換必須分塊** — 不同人的發言不能合在同一區塊

#### 區塊合併思考邏輯（重要！）

在重組之前，必須思考：

1. **原始 SRT 有幾個區塊？** — 這決定了你可用的時間點數量
2. **校正後需要幾個區塊？** — 根據語意斷句決定
3. **長度平衡** — 不能太短（如只有 5 個字），也不能太長（超過 50 字）

| 情況 | 處理方式 |
|------|----------|
| 原始區塊太碎（如每 2 秒一塊） | 合併多個原始區塊成一個語意完整的句子 |
| 原始區塊太長（如 30 秒一塊） | 在原始時間範圍內，根據字數比例分配時間點 |
| 校正後區塊數 > 原始時間點數 | 必須減少區塊數或合併句子 |

#### 時間碼對應方法

1. **列出所有原始時間點** — 這是你唯一可用的時間來源
2. **規劃區塊分配** — 校正後區塊數不應超過原始時間點數的 2-3 倍
3. **對應方式**：
   - 如果一句話跨越多個原始區塊，取「起始時間」和「結束時間」
   - 如果一個原始區塊包含多句話，根據字數比例內插時間
   - **禁止**：憑空創造不存在於原始 SRT 的時間點

#### 區塊長度指南

```
太短（避免）：
   「好。」「對。」「然後。」

適中（理想）：
   「我們來看向創的評分模型，目前用 GPT-4o-mini 可以一次評 60 個屬性。」
   「曼巴的資料都放在 DuckDB 上面，有問題可以直接來看。」

太長（拆分）：
   「反正就是因為它原本一個個評要很慢然後我就做了一些測試
   用 GPT-5 mini 然後它就可以一則評論直接評 60 個屬性
   我就用 Claude 做了一個研究...」→ 應拆成 2-3 個區塊
```

### Step 4：輸出校正後的 SRT

1. **每個句子一個時間戳** — 以句點為分界
2. 時間碼從原始 SRT 對應而來
3. 適當加入標點符號（逗號、句點）
4. 儲存位置與命名：
   - 同目錄下，檔名加 `_corrected` 後綴
   - 格式：`YYYYMMDD_主題-transcript_corrected.srt`

### Step 5：歸檔原始 SRT

校正完成後，將原始（未校正）的 SRT 移至同目錄的 `archived/`：

```
mv YYYYMMDD_主題-transcript.srt archived/
```

- `archived/` 同時存放歸檔的 MP3 音檔和校正前的原始 SRT
- 校正後的 `_corrected.srt` 留在 records 根目錄作為正式版本

## 校正原則

### 保留原則
- 保留說話者的表達風格
- 保留必要的語氣詞（如確認用的「對」、轉折用的「好，接下來...」）
- 保留會議互動（如「等一下喔」「你分享給我」）
- 保留非正式但有意義的討論內容

### 修改原則
- 優先校正公司名和產品名（最影響可讀性）
- 技術術語以專案實際用法為準
- 口語轉書面時保持自然，不要過度正式化
- 會議記錄不同於講稿，允許更口語的風格

### 不處理原則
- **不要刪除「閒聊」段落** — 會議中的非正式討論可能包含重要決策
- **不要重寫句子** — 目標是讓字幕可讀，不是重寫會議紀要
- **不要推測未說出的內容** — 聽不清楚用 `[?]` 標記

## 範例

### 範例 1：公司名校正 + 去贅詞

**輸入（原始）：**
```srt
2
00:00:14,430 --> 00:00:47,530
che: 對 然後就是評分的那個部分我有做一點研究 就是說它可以就是用, 就我做了一些測試 然後就用GPT-5的mini 然後它就可以一則評論直接評60個, 60個屬性, 我就是我做了一個研究 就我用Cloud做一個研究 就以Cloud它自己的評分為標準.
```

**輸出（校正後）：**
```srt
2
00:00:14,430 --> 00:00:30,000
che: 評分的部分我有做一點研究，做了一些測試。用 GPT-5 mini 可以一則評論直接評 60 個屬性。

3
00:00:30,000 --> 00:00:47,530
che: 我用 Claude 做了一個研究，以 Claude 自己的評分為標準。
```

> 注意：「GPT-5的mini」校正為「GPT-5 mini」（去掉「的」），而非改成其他模型名。
> 「Cloud」校正為「Claude」（同音異字）。

### 範例 2：說話者切換

**輸入（原始）：**
```srt
51
00:03:12,828 --> 00:03:18,448
che: 我現在測試你那個相創是不是還沒有主題.
```

**輸出（校正後）：**
```srt
51
00:03:12,828 --> 00:03:18,448
che: 我現在測試你那個向創是不是還沒有主題。
```

### 範例 3：錯誤示範（虛構時間）

```srt
1
00:00:00,000 --> 00:00:02,145   ← 2,145 從哪來？原始沒有這個時間點！
che: 好，那你要不要等昊紘上來。
```

> 這種「平均分配」的時間碼是錯誤的，必須使用原始 SRT 的實際時間點。

## 注意事項

1. **每句一戳**：一個完整語意的句子對應一個時間戳
2. **時間碼絕對禁止虛構**：
   - 正確：從原始 SRT 的區塊衍生時間點
   - 錯誤：憑空創造不存在的時間
   - 嚴重錯誤：校正後時間超過原始錄音長度
3. **區塊數量檢查**：校正後區塊數應與原始在合理比例內（約 1-3 倍）
4. **長度平衡**：每個區塊 30-50 字
5. **說話者標籤**：每個區塊必須有說話者標籤，不同說話者不合併
6. **標記不確定處**：用 `[?]` 標記聽不清楚的地方
7. **檔名規範**：輸出檔名遵循 `YYYYMMDD_主題-transcript_corrected.srt`
