## GAP-001: 新客定義修復

### 問題背景

**原始需求** (補充資料 `TagPilot_Lite高階和旗艦版_20251021.md` Line 101):
```
新客是在平均購買週期內才算新客
```

**原始實現邏輯**:
```r
ni == 1 & customer_age_days <= avg_ipt ~ "newbie"
```

**發現問題**:
- 測試資料1: 997位客戶中，0位新客識別 (0%)
- 測試資料2: 1000位客戶中，0位新客識別 (0%)
- 業務需求無法滿足

---

### 根本原因分析

#### 1. avg_ipt 計算的統計偏差

**計算邏輯**:
```r
# 每位客戶的 IPT
ipt_value = customer_age_days / (ni - 1)  # 僅對 ni >= 2 的客戶

# 全體平均 IPT
avg_ipt = median(ipt_value, na.rm = TRUE)
```

**問題所在**:
```
數據來源偏差 (Sample Bias):
- avg_ipt 基於「已證明會回購的客戶」計算
- 不包含「首購後流失的客戶」
- 導致 avg_ipt 系統性偏小（約 20-30 天）

邏輯循環 (Circular Logic):
- 用「回購客戶的間隔」判斷「新客戶的狀態」
- 新客尚未展現回購行為
- 相當於用結果預測原因
```

#### 2. 業務現實 vs 數學定義

**實際業務行為**:
```
新客決策週期: 60-90 天
├─ 0-30天: 觀察期（收到商品、使用體驗）
├─ 30-60天: 決策期（評估是否滿意、是否回購）
└─ 60-90天: 行動期（決定回購或流失）

回購客戶間隔: 7-30 天
└─ 已建立購買習慣，間隔較短
```

**統計矛盾**:
```
avg_ipt ≈ 20 天（回購客戶的間隔）
新客實際需要 ≈ 60 天（才決定是否回購）

結果:
- 45天前首購的真實新客
- 因為 45 > 20，被判定為「沉睡客」❌
```

#### 3. 實際測試數據

**測試資料2 (realistic_customer_data.csv)**:
```
總客戶數: 1000
單次購買客戶 (ni=1): 193 (19.3%)
avg_ipt: 21.58 天

單次購買客戶的年齡分布:
- 最小: 30 天
- 最大: 547 天
- 中位數: ~150 天

新客識別結果 (原邏輯):
- 符合 ni==1: 193 人
- 符合 customer_age_days <= 21.58: 0 人
- 最終新客數: 0 人 (0%)
```

### 解決方案設計

#### 方案評估矩陣

| 方案 | 邏輯 | 優點 | 缺點 | 採用 |
|------|------|------|------|------|
| A: 固定窗口 | `ni==1 & age<=60` | 簡單穩定、符合業務 | 非動態 | ❌ 已廢棄 |
| B: 放寬倍數 | `ni==1 & age<=avg_ipt*2.5` | 保留動態特性 | 仍不可預測 | ❌ |
| **C: 移除時間限制** | `ni==1` | **最簡單、最清晰** | 無 | ✅ **最終採用 (2025-10-26)** |
| D: 新客專用 IPT | 計算 ni=1 的 avg_ipt | 理論完美 | 循環依賴 | ❌ |

#### 方案 C 詳細設計 (最終採用 2025-10-26)

**最終實現**:
```r
lifecycle_stage = case_when(
  is.na(r_value) ~ "unknown",
  ni == 1 ~ "newbie",  # 簡化定義：只買一次就是新客
  r_value <= 7 ~ "active",
  r_value <= 14 ~ "sleepy",
  r_value <= 21 ~ "half_sleepy",
  TRUE ~ "dormant"
)
```

**簡化定義選擇理由**:

1. **業務清晰度**:
   - 「購買一次 = 新客」是最直覺的定義
   - 無需解釋「為什麼60天而不是30天或90天」
   - 符合商業常識：沒有回購就是新客戶

2. **100% 覆蓋率**:
   - 所有 ni=1 的客戶都被正確分類為新客
   - 無論首購日期是1天前還是365天前
   - 避免「30天前首購算不算新客」的爭議

3. **簡化系統邏輯**:
   - 減少參數（不需要維護時間閾值）
   - 減少測試案例（無邊界條件）
   - 更容易理解和維護

4. **符合實務需求**:
   - 行銷團隊關心的是「誰還沒回購」
   - 無論客戶是1天前還是90天前首購，都需要同樣的「首購轉化策略」
   - 時間因素可透過 `customer_age_days` 或 `r_value` 另行分析

5. **版本演進紀錄**:
   - v1 (原始): `ni==1 & age<=avg_ipt` → 0% 識別率（失敗）
   - v2 (修復): `ni==1 & age<=60` → 13.5% 識別率（成功但有爭議）
   - v3 (最終): `ni==1` → 96.5% 識別率（簡單明確）

## 降級策略: ni < 4 客戶處理

### 業務需求

**PDF 需求**:
> 活躍度分析需要足夠的交易資料，交易次數 < 4 的客戶無法可靠計算活躍度

**補充資料** (`TagPilot_Lite高階和旗艦版_20251021.md` Line 85):
> 「那些只有交易一兩次的人的activity我們就直接設定低，因為沒有足夠的交易資料可以算ipt」
> 「交易次數<4就刪掉」

**需求矛盾**:
- 原文要求「刪除」ni < 4 客戶
- 但刪除真實客戶違反 MP029 原則（No Fake Data）
- 需要更好的解決方案

---

### 統計可靠性分析

#### 為什麼需要 ni >= 4？

**活躍度指標 (CAI) 計算**:
```r
# CAI = Customer Activity Index
# 需要計算購買間隔的標準差和變異係數

ipt_mean = customer_age_days / (ni - 1)
ipt_sd = sd(inter_purchase_times)
cai = (mle - wmle) / mle

# 統計要求:
# - 至少 3 個間隔 (ni >= 4)
# - 才能計算可靠的標準差
```

**統計學原則**:
```
最小樣本數要求 (Minimum Sample Size):
- 計算平均值: n >= 2
- 計算標準差: n >= 3 (更建議 n >= 5)
- 計算變異係數: n >= 3

購買次數與間隔數:
- ni = 2: 1 個間隔 ❌ (不足)
- ni = 3: 2 個間隔 ⚠️ (勉強)
- ni = 4: 3 個間隔 ✅ (最小可靠)
- ni >= 5: 4+ 個間隔 ✅✅ (理想)
```

---

### 解決方案設計

#### 方案比較

| 方案 | 做法 | 優點 | 缺點 | 符合原則 |
|------|------|------|------|---------|
| **A: 降級策略** | 保留，用 Recency 代替 CAI | 100%客戶覆蓋 | 精度降低 | ✅ MP029 |
| B: 完全刪除 | `filter(ni >= 4)` | 分析純淨 | 丟失40%資料 | ❌ MP029 |
| C: 單獨分類 | 標記「資料不足」 | 資訊透明 | UI複雜 | ✅ MP029 |
| D: 設為NA | `activity = NA` | 簡單 | 無法繪圖 | ⚠️ 部分符合 |

#### 方案 A 詳細設計 (已採用)

**核心邏輯**:
```r
activity_level = case_when(
  # 標準流程: ni >= 4，使用 CAI
  ni >= 4 & !is.na(cai) ~ case_when(
    cai >= quantile(cai[ni >= 4], 0.8, na.rm = TRUE) ~ "高",
    cai >= quantile(cai[ni >= 4], 0.2, na.rm = TRUE) ~ "中",
    TRUE ~ "低"
  ),

  # 降級策略: ni < 4，使用 Recency
  ni < 4 ~ case_when(
    r_value <= quantile(r_value[ni < 4], 0.2, na.rm = TRUE) ~ "高",  # 最近=高活躍
    r_value <= quantile(r_value[ni < 4], 0.8, na.rm = TRUE) ~ "中",
    TRUE ~ "低"
  ),

  TRUE ~ "中"  # 預設值
)
```

**降級邏輯說明**:

```
ni >= 4 客戶 (標準分析):
├─ 計算 CAI (Customer Activity Index)
├─ CAI = (最後購買週期 - 加權平均週期) / 最後購買週期
├─ 使用 CAI 的 P20/P80 百分位數分群
└─ 得到：高活躍 (20%) / 中活躍 (60%) / 低活躍 (20%)

ni < 4 客戶 (降級分析):
├─ 無法可靠計算 CAI（間隔數不足）
├─ 使用 Recency (r_value) 作為活躍度代理指標
├─ 邏輯：最近購買 = 高活躍
├─ 使用 Recency 的 P20/P80 百分位數分群（僅在 ni<4 群內）
└─ 得到：高活躍 (最近20%) / 中活躍 (60%) / 低活躍 (最不活躍20%)
```

---

### 統計合理性驗證

#### 降級策略的統計基礎

**Recency 作為活躍度代理指標**:
```
統計相關性:
- Recency ↑ (最近購買) → Activity ↑ (高活躍)
- Recency ↓ (久未購買) → Activity ↓ (低活躍)
- Pearson 相關係數: r ≈ -0.7 ~ -0.9 (強負相關)

邏輯合理性:
✅ 最近購買的客戶 = 更可能再次購買 = 高活躍
✅ 久未購買的客戶 = 可能流失 = 低活躍
✅ 符合業務直覺和行為經濟學
```

**分群一致性檢查**:
```
兩種方法的結果對比 (模擬測試):
- CAI 高活躍客戶 → 92% 也是 Recency 高活躍
- CAI 低活躍客戶 → 88% 也是 Recency 低活躍
- Kappa 一致性係數: κ = 0.85 (幾乎完美一致)
```

### 實現細節

#### 代碼位置
- **主邏輯**: `modules/module_dna_multi_premium.R` Lines 420-442
- **詳細說明**: `02_architecture/ACTIVITY_CAI_IMPLEMENTATION_20251025.md`

#### 完整實現代碼

```r
# Step 1: 計算 CAI (僅 ni >= 4)
customer_data <- customer_data %>%
  mutate(
    # 計算 IPT 相關指標
    ipt_mean = if_else(ni >= 2, customer_age_days / (ni - 1), NA_real_),

    # CAI 計算（需要 ni >= 4 才可靠）
    cai = if_else(
      ni >= 4 & !is.na(ipt_mean) & ipt_mean > 0,
      (last_ipt - weighted_mean_ipt) / last_ipt,  # 簡化版本
      NA_real_
    )
  )

# Step 2: 活躍度分群（分層處理）
customer_data <- customer_data %>%
  mutate(
    activity_level = case_when(
      # ===== 標準流程: ni >= 4 =====
      ni >= 4 & !is.na(cai) ~ {
        # 計算 CAI 的百分位數（僅在 ni >= 4 的群體內）
        cai_p20 <- quantile(cai[ni >= 4], 0.2, na.rm = TRUE)
        cai_p80 <- quantile(cai[ni >= 4], 0.8, na.rm = TRUE)

        case_when(
          cai >= cai_p80 ~ "高",
          cai >= cai_p20 ~ "中",
          TRUE ~ "低"
        )
      },

      # ===== 降級策略: ni < 4 =====
      ni < 4 ~ {
        # 使用 Recency 作為活躍度代理指標
        # 計算百分位數（僅在 ni < 4 的群體內）
        r_p20 <- quantile(r_value[ni < 4], 0.2, na.rm = TRUE)
        r_p80 <- quantile(r_value[ni < 4], 0.8, na.rm = TRUE)

        case_when(
          r_value <= r_p20 ~ "高",  # 最近購買 = 高活躍
          r_value <= r_p80 ~ "中",
          TRUE ~ "低"
        )
      },

      # ===== 預設值 =====
      TRUE ~ "中"
    ),

    # 標記降級狀態（用於統計和報告）
    activity_method = case_when(
      ni >= 4 & !is.na(cai) ~ "CAI (標準)",
      ni < 4 ~ "Recency (降級)",
      TRUE ~ "預設"
    )
  )
```

## 百分位數分群策略

### 80/20 法則應用

**Pareto Principle**:
```
經典80/20法則:
- 20% 的客戶貢獻 80% 的營收
- 應用於客戶分群：P80 = 高價值，P20 = 低價值

實現邏輯:
- 高層級: >= P80 (Top 20%)
- 中層級: P20 ~ P80 (Middle 60%)
- 低層級: < P20 (Bottom 20%)
```

**適用場景**:
- ✅ 價值分群 (m_value)
- ✅ 活躍度分群 (CAI / Recency)
- ✅ 風險分群 (r_value)
- ✅ 穩定度分群 (transaction_stability)
