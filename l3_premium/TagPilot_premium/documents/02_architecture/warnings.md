### SW-001: 新客定義的統計意義 🟡

**警告類型**: 統計假設變更
**影響範圍**: Module 1（DNA 分析）、Module 4（生命週期）
**嚴重程度**: 🟡 中等
**狀態**: ✅ 已說明並文檔化

#### 問題描述

**原始定義**（基於補充資料）：
```r
新客 = ni == 1 & customer_age_days <= avg_ipt
```

**統計邏輯**：
- `avg_ipt` 基於**回購客戶**（ni ≥ 2）計算
- 假設：新客應在「回購客戶的平均回購週期」內回購
- **問題**：這個假設在統計上不成立

**邏輯矛盾**：
1. **樣本偏差**：
   - avg_ipt 樣本 = 已回購客戶（成功案例）
   - 新客群體 = 首次購買，包含成功 + 流失案例
   - 用成功案例的平均值判斷整體 → 標準過於嚴格

2. **時間悖論**：
   - 回購客戶 IPT ≈ 20-30 天（已建立購買習慣）
   - 新客決策週期 ≈ 60-90 天（首次考察期）
   - 60天首購客戶 > 20天 avg_ipt → 被誤判為沉睡客

3. **統計結果**：
   ```
   測試資料驗證：
   - avg_ipt = 21.58 天
   - 符合原定義的新客 = 0 人（0%）❌
   - 實際首購 < 60 天 = 135 人（13.5%）✅
   ```

#### 修正方案

**採用固定窗口**：
```r
新客 = ni == 1 & customer_age_days <= 60
```

**統計合理性**：
1. ✅ **符合業務現實**：60 天是標準新客觀察期
2. ✅ **統計穩定性**：不受資料分佈波動影響
3. ✅ **行業標準**：多數 CRM 系統使用 30-90 天固定窗口
4. ✅ **可預測性**：不會因新增資料導致定義改變

#### 統計意義說明

| 指標 | 原定義（avg_ipt） | 修正定義（60天） |
|------|-----------------|----------------|
| **統計邏輯** | 動態（依資料變化） | 固定（業務定義） |
| **樣本偏差** | 高（僅回購客戶） | 無 |
| **穩定性** | 低（隨資料波動） | 高（固定參數） |
| **可解釋性** | 複雜（需解釋 avg_ipt） | 簡單（60天易懂） |
| **業務適用性** | 低（0% 識別率） | 高（13.5% 識別率） |

#### 使用建議

1. **數據解讀**：
   - 新客比例 5-15% 為正常範圍
   - < 5%：可能新客流失率高，需加強首購體驗
   - > 20%：可能業務處於高速成長期

2. **策略應用**：
   - 60 天內新客：首購體驗優化、二購誘因
   - 60-90 天首購客戶：雖非「新客」標籤，仍需特殊關注

3. **參數調整**：
   - 可在 UI 設定中提供 30/60/90 天選項
   - 建議：B2C 電商用 60 天，B2B 用 90 天

**詳細分析**: [GAP001_NEWBIE_DEFINITION_ANALYSIS.md](GAP001_NEWBIE_DEFINITION_ANALYSIS.md)

### SW-002: 活躍度降級策略的統計意義 🟢

**警告類型**: 指標替換
**影響範圍**: Module 3（活躍度分析）、Module 5（九宮格）
**嚴重程度**: 🟢 低（已妥善處理）
**狀態**: ✅ 已實現並文檔化

#### 問題描述

**統計要求**（基於補充資料 Line 85）：
```
理論上,交易次數少於四次的顧客沒辦法算活躍度,故將低於四次交易的顧客刪除。
```

**統計原理**：
- CAI（Customer Activity Index）需要至少 4 筆交易計算趨勢
- 少於 4 筆無法有效估計購買行為的加速/減速

**問題**：刪除 ni < 4 的客戶違反 MP029（No Fake Data）

#### 降級策略實現

**技術方案**：使用 **Recency 替代 CAI**

```r
activity_level = case_when(
  # 標準計算：ni ≥ 4，使用 CAI
  ni >= 4 & !is.na(cai) ~ case_when(
    cai >= quantile(cai[ni >= 4], 0.8, na.rm = TRUE) ~ "高",
    cai >= quantile(cai[ni >= 4], 0.2, na.rm = TRUE) ~ "中",
    TRUE ~ "低"
  ),

  # 降級計算：ni < 4，使用 Recency
  ni < 4 ~ case_when(
    r_value <= quantile(r_value[ni < 4], 0.2, na.rm = TRUE) ~ "高",
    r_value <= quantile(r_value[ni < 4], 0.8, na.rm = TRUE) ~ "中",
    TRUE ~ "低"
  ),

  TRUE ~ "中"
)
```

#### 統計意義對照

| 客戶類型 | 活躍度指標 | 統計意義 | 百分位數計算 |
|---------|-----------|----------|-------------|
| **ni ≥ 4** | CAI | 購買行為**趨勢**（加速/減速） | P20/P80（CAI 分佈） |
| **ni < 4** | Recency | 當前購買**狀態**（近期/久遠） | P20/P80（R 值分佈） |

**關鍵差異**：
- **CAI（ni ≥ 4）**：
  - 衡量：購買頻率是否**增加**或**減少**
  - 範例：從 30天一次 → 15天一次（加速，高活躍）

- **Recency（ni < 4）**：
  - 衡量：最近一次購買是否**近期**
  - 範例：7天前購買（近期，高活躍）

#### 統計合理性評估

| 評估項目 | 刪除方案 | 降級方案（採用） | 統計評分 |
|---------|---------|-----------------|---------|
| **樣本完整性** | 60% | 100% | ✅ 降級優 |
| **CAI 準確度** | 100% | 100%（ni≥4） | ✅ 相同 |
| **活躍度語義** | 統一（趨勢） | 分層（趨勢+狀態） | ⚠️ 需說明 |
| **業務適用性** | 部分客戶缺失 | 全客戶覆蓋 | ✅ 降級優 |
| **MP029 合規** | ❌ 違反 | ✅ 符合 | ✅ 降級優 |

#### 使用建議

1. **數據解讀**：
   ```
   活躍度分佈：
   - 高活躍：150 人（15%）
   - 中活躍：600 人（60%）
   - 低活躍：250 人（25%）

   註：交易次數 ≥4 使用 CAI（趨勢指標），
       交易次數 <4 使用 Recency（狀態指標）
   ```

2. **策略應用**：
   - **ni ≥ 4 高活躍**：購買加速中，推薦新品、會員升級
   - **ni < 4 高活躍**：最近有購買，推薦回購或搭售

3. **報告呈現**：
   - CSV 導出新增欄位：`activity_calculation_method`
   - 值：`CAI`（ni≥4）或 `Recency`（ni<4）
   - 便於使用者理解

#### 統計警告

⚠️ **警告**：
- ni < 4 的「高活躍」≠ ni ≥ 4 的「高活躍」
- 前者僅表示「最近有購買」，不代表「購買頻率增加」
- 策略應用時需區分語義

✅ **優勢**：
- 所有客戶都有活躍度標籤
- 更完整的客戶視圖
- 符合 MP029（保留真實資料）

**詳細分析**: [COMPREHENSIVE_COMPLIANCE_AUDIT_20251026.md](COMPREHENSIVE_COMPLIANCE_AUDIT_20251026.md) (DIFF-002)

## 🔴 C001: 新客定義導致0新客問題

### 問題描述

**原始邏輯**：
```r
lifecycle_stage <- case_when(
  customer_age_days <= 30 ~ "newbie",  # ❌ 問題所在
  r_value <= 7 ~ "active",
  r_value <= 14 ~ "sleepy",
  r_value <= 21 ~ "half_sleepy",
  TRUE ~ "dormant"
)
```

**問題表現**：
- 某些業態（如B2B、高單價商品）平均購買週期 > 30天
- 客戶在30天內購買後，若未在7天內再次購買，直接被分類為睡眠客
- 導致系統中出現 **0個新客** 的異常現象

**案例**：
```
客戶A:
- 首次購買日期: 2025-10-01
- 今天: 2025-10-25
- customer_age_days = 24天
- times = 1
- r_value = 24天

原邏輯判斷流程:
1. customer_age_days <= 30? YES → 新客 (N)  ❌ 錯誤！
   （實際上，case_when 遇到第一個 TRUE 就停止）

實際判斷流程:
1. customer_age_days <= 30? YES → 應該是新客
2. 但由於 r_value = 24 > 21，後續條件全不符
3. 最終被判為 dormant（沉睡客）❌

問題根源: case_when 的順序導致新客被錯誤分類
```

### 解決方案

**修正後邏輯**：
```r
lifecycle_stage <- case_when(
  times == 1 & customer_age_days <= avg_ipt ~ "newbie",  # ✅ 修正
  r_value <= 7 ~ "active",
  r_value <= 14 ~ "sleepy",
  r_value <= 21 ~ "half_sleepy",
  TRUE ~ "dormant"
)
```

**修正說明**：
1. **增加 `times == 1` 條件**：確保只有首次購買的客戶才是新客
2. **使用 `avg_ipt` 替代固定30天**：動態適應不同業態的購買週期
3. **邏輯更嚴謹**：新客定義現在明確且合理

**修正效果**：
- ✅ 新客數量恢復正常
- ✅ 適應不同業態（avg_ipt 會根據實際資料計算）
- ✅ 避免誤判（首次購買 + 未超過平均週期）

### 影響範圍

- ✅ 修正檔案：`module_dna_multi_premium.R`
- ✅ 受影響功能：生命週期分類、九宮格分析
- ✅ 向後兼容：需要重新執行舊資料分析

## 🟡 C002: 活躍度計算門檻與新客衝突

### 問題描述

**需求矛盾**：
1. **活躍度計算要求**：交易次數 >= 4
2. **新客定義**：`times == 1` (首次購買)

**衝突表現**：
```r
# 新客判斷
if (times == 1 & customer_age_days <= avg_ipt) {
  lifecycle_stage = "newbie"
}

# 活躍度計算
data_with_activity <- dna_data %>%
  filter(times >= 4)  # ❌ 新客被排除！
```

**結果**：
- 所有新客（times = 1）都不符合活躍度計算條件（times >= 4）
- 新客無法被分配到九宮格中
- 隱藏的6種新客組合（A1N, A2N, B1N, B2N, C1N, C2N）實際上 **不可能出現**

### 影響分析

| 客戶類型 | times | 能否計算活躍度? | 能否進入九宮格? |
|---------|-------|---------------|---------------|
| 新客 (N) | 1 | ❌ 不符合 >= 4 | ❌ 被排除 |
| 2-3次購買客戶 | 2-3 | ❌ 不符合 >= 4 | ❌ 被排除 |
| 成熟客戶 | >= 4 | ✅ 符合 | ✅ 正常分析 |

### 潛在解決方案

#### 方案A: 新客獨立處理（建議）

```r
# 新客單獨分群（不計算活躍度）
newbie_customers <- dna_data %>%
  filter(times == 1 & customer_age_days <= avg_ipt) %>%
  mutate(
    grid_position = "NEWBIE",
    strategy = "新客歡迎計畫"
  )

# 成熟客戶正常九宮格分析
mature_customers <- dna_data %>%
  filter(times >= 4) %>%
  # ... 正常九宮格邏輯
```

**優點**：
- ✅ 邏輯清晰

## 🟢 C004: RFM vs M-only 價值計算爭議

### 問題描述

**當前實作**：
```r
# 價值層級僅使用 M 值
value_level <- case_when(
  m_value >= quantile(m_value, 0.8, na.rm = TRUE) ~ "高",
  m_value >= quantile(m_value, 0.2, na.rm = TRUE) ~ "中",
  TRUE ~ "低"
)
```

**爭議點**：
- 是否應該考慮 R 值（近期活躍度）？
- 是否應該考慮 F 值（購買頻率）？
- 「價值」的定義：總貢獻 vs 當前潛力？

### 不同方法比較

| 方法 | 計算方式 | 優點 | 缺點 | 適用場景 |
|-----|---------|------|------|---------|
| **M-only**<br>(當前) | 僅總消費金額 | 簡單明確<br>歷史貢獻真實 | 忽略近期狀態<br>可能高估流失客戶 | 累積價值評估 |
| **RFM綜合** | M × R × F 加權 | 全面考量<br>反映當前狀態 | 計算複雜<br>權重難定義 | 當前潛力評估 |
| **M + R調整** | M × decay_factor(R) | 考慮近期活躍<br>計算可控 | 需定義衰減函數 | 動態價值評估 |

### 案例分析

**案例1: 高M值但已流失的客戶**

```
客戶A:
- M值 = 50,000元（過去累積）
- R值 = 180天（半年未購買）
- F值 = 20次

M-only 判斷: 高價值 ✅
RFM綜合判斷: 中/低價值（因R值太高）❌

業務觀點: 這是需要喚醒的高價值流失客戶
```

**結論**: M-only 在此案例中更合理（識別出高價值流失客戶）

**案例2: 低M值但高潛力的活躍客戶**

```
客戶B:
- M值 = 2,000元（剛開始購買）
- R值 = 2天（非常活躍）
- F值 = 8次（三個月內）

M-only 判斷: 低價值 ❌
RFM綜合判斷: 中/高價值（因R、F都很好）✅

業務觀點: 這是成長潛力客戶
```

**結論**: RFM綜合在此案例中更合理（識別出潛力客戶）

### 建議解決方案

**採用雙重價值計算**：

```r
# 1. 歷史價值（M-only）- 用於識別貢獻客戶
historical_value <- case_when(
  m_value >= quantile(m_value, 0.8) ~ "高",
  m_value >= quantile(m_value, 0.2) ~ "中",
  TRUE ~ "低"
)

# 2. 當前潛力（RFM綜合）- 用於識別機會客戶
current_potential <- calculate_rfm_score(r, f, m)

# 3. 策略組合
grid_position_history <- paste0(historical_value, activity, lifecycle)
grid_position_potential <- paste0(current_potential, activity, lifecycle)
```

**優點**：
- ✅ 兩種視角並存
- ✅ 更全面的客戶洞察
- ✅ 支援不同業務場景

**缺點**：
- ⚠️ UI需要支援雙視圖切換
- ⚠️ 策略建議需要區分兩種場景

### 當前狀態

💭 **設計討論中**

**暫行做法**：
- 保持 M-only（符合「Value × Activity」命名）
- 在文件中說明這是「累積價值」視角
- 為未來雙重視角預留擴展空間

## 🟡 C006: 百分位數計算樣本數不足

### 問題描述

**百分位數方法依賴足夠樣本**：
```r
value_level <- case_when(
  m_value >= quantile(m_value, 0.8, na.rm = TRUE) ~ "高",
  m_value >= quantile(m_value, 0.2, na.rm = TRUE) ~ "中",
  TRUE ~ "低"
)
```

**問題表現**：

| 客戶數 | P20 | P80 | 分群結果 | 問題 |
|-------|-----|-----|---------|------|
| 1000+ | 穩定 | 穩定 | 正常 20/60/20 | ✅ 無問題 |
| 100 | 較穩定 | 較穩定 | 接近 20/60/20 | ⚠️ 可接受 |
| 50 | 不穩定 | 不穩定 | 可能偏離 | ⚠️ 需警告 |
| 10 | 極不穩定 | 極不穩定 | 嚴重偏離 | ❌ 不建議使用 |
| 5 | 無意義 | 無意義 | 完全失真 | ❌ 禁止使用 |

**案例**：
```
5個客戶的M值: [1000, 2000, 3000, 4000, 5000]

P80 = quantile(M, 0.8) = 4200
P20 = quantile(M, 0.2) = 1800

分群結果:
- 高價值: M >= 4200 → 僅客戶5 (20%) ✅ 看似正常
- 中價值: 1800 <= M < 4200 → 客戶2,3,4 (60%) ✅ 看似正常
- 低價值: M < 1800 → 僅客戶1 (20%) ✅ 看似正常

問題: 僅5個樣本，百分位數沒有統計意義
```

### 解決方案

**增加樣本數檢查**：

```r
# 在計算前檢查樣本數
MIN_SAMPLE_SIZE <- 30  # 統計學最小樣本數建議

if (nrow(data_for_activity) < MIN_SAMPLE_SIZE) {
  showNotification(
    paste0(
      "警告：客戶數量不足（", nrow(data_for_activity), "），",
      "建議至少需要 ", MIN_SAMPLE_SIZE, " 個客戶才能進行可靠的分群分析。",
      "當前分析結果僅供參考。"
    ),
    type = "warning",
    duration = 10
  )
}

# 樣本數太少時，使用絕對值判斷
if (nrow(data_for_activity) < MIN_SAMPLE_SIZE) {
  # 改用固定閾值而非百分位數
  value_level <- case_when(
    m_value >= 10000 ~ "高",
    m_value >= 2000 ~ "中",
    TRUE ~ "低"
  )
} else {
  # 樣本數足夠，使用百分位數
  value_level <- case_when(
    m_value >= quantile(m_value, 0.8, na.rm = TRUE) ~ "高",
    m_value >= quantile(m_value, 0.2, na.rm = TRUE) ~ "中",
    TRUE ~ "低"
  )
}
```

### 建議最小樣本數

| 分析類型 | 最小樣本數 | 建議樣本數 | 理由 |
|---------|-----------|-----------|------|
| 九宮格分析 | 30 | 100+ | 統計學基本要求 |
| 百分位數計算 | 50 | 200+ | 確保分布穩定 |
| 活躍度計算 | 20 | 50+ | 需要足夠變異 |
| 生命週期預測 | 100 | 500+ | 機器學習模型需求 |

### 當前狀態

⚠️ **需要實作**

**建議**：
1. 在 UI 顯示當前樣本數
2. 樣本數不足時顯示警告
3. 提供「樣本數說明」文件給用戶

## 🟢 C007: 生命週期階段臨界值爭議

### 問題描述

**當前臨界值**：
```r
lifecycle_stage <- case_when(
  times == 1 & customer_age_days <= avg_ipt ~ "newbie",
  r_value <= 7 ~ "active",      # 7天
  r_value <= 14 ~ "sleepy",     # 14天
  r_value <= 21 ~ "half_sleepy", # 21天
  TRUE ~ "dormant"
)
```

**問題**：
- 7/14/21 天是否適用所有業態？
- B2C vs B2B 購買週期差異巨大
- 高頻商品（如咖啡）vs 低頻商品（如家電）

### 不同業態購買週期參考

| 業態 | 平均購買週期 | 建議臨界值 (R) | 說明 |
|-----|-------------|---------------|------|
| 快銷品（咖啡、飲料） | 3-7天 | 3/7/14 | 需要更短週期 |
| 一般零售 | 7-30天 | 7/14/21 | 當前設定 ✅ |
| 服飾 | 30-90天 | 30/60/90 | 需要更長週期 |
| 家電 | 180-365天 | 90/180/270 | 大幅延長週期 |
| B2B | 30-180天 | 30/60/90 | 視產業而定 |

### 解決方案

#### 方案A: 動態臨界值（建議）

```r
# 根據平均IPT動態調整
avg_ipt <- median(dna_data$avg_ipt, na.rm = TRUE)

# 臨界值設為 avg_ipt 的倍數
threshold_active <- avg_ipt * 0.5
threshold_sleepy <- avg_ipt * 1.0
threshold_half_sleepy <- avg_ipt * 1.5

lifecycle_stage <- case_when(
  times == 1 & customer_age_days <= avg_ipt ~ "newbie",
  r_value <= threshold_active ~ "active",
  r_value <= threshold_sleepy ~ "sleepy",
  r_value <= threshold_half_sleepy ~ "half_sleepy",
  TRUE ~ "dormant"
)
```

**優點**：
- ✅ 自動適應不同業態
- ✅ 基於實際資料
- ✅ 邏輯一致性高

**缺點**：
- ⚠️ 需要足夠樣本計算可靠的avg_ipt
- ⚠️ 極端值可能影響結果

#### 方案B: 可配置參數


**缺點**：
