# analyze_customer_dynamics_new.R 函數說明文件

## 檔案資訊

- **檔案路徑**: `utils/analyze_customer_dynamics_new.R`
- **功能**: Z-Score 統計方法計算客戶動態
- **最後更新**: 2025-11

---

## 函數概述

`analyze_customer_dynamics_new()` 是整個系統的**核心分析引擎**，主要做 **5 大分析**：

1. 產業基準計算（μ_ind、W）
2. 客戶活躍度指標（F_i,w、z_i）
3. 客戶動態分類（newbie/active/sleepy/half_sleepy/dormant）
4. 價值等級分類（高/中/低）
5. 基礎 RFM 指標

---

## 核心分析內容

### 1. 產業基準計算 (Step 2-3)

#### 計算 μ_ind（產業平均購買間隔）

```r
# 計算所有回購客戶的購買間隔，取中位數
μ_ind = median(所有客戶的購買間隔)

# 例如：μ_ind = 30 天（表示平均每 30 天會回購）
```

#### 計算 W（活躍觀察窗）

```r
W = min(觀察期, max(90天, round_to_7(2.5 × μ_ind)))

# 例如：μ_ind = 30 → W = 84 天（12 週）
```

---

### 2. 客戶活躍度指標 (Step 4-7)

#### 計算 F_i,w（最近 W 天內的購買次數）

```r
# 統計每位客戶在最近 W 天內買了幾次
客戶A：最近 84 天買了 5 次 → F_i,w = 5
客戶B：最近 84 天買了 1 次 → F_i,w = 1
```

#### 計算 z_i（標準化活躍度分數）

```r
λ_w = mean(F_i,w)     # 平均購買次數
σ_w = sd(F_i,w)       # 標準差
z_i = (F_i,w - λ_w) / σ_w

# 解讀：
# z_i > 0.5  → 活躍度高於平均（主力客）
# z_i < -1.0 → 活躍度遠低於平均（瞌睡客）
```

---

### 3. 客戶動態分類 (Step 8) ⭐

使用兩種方法之一：

#### 方法 A：Z-Score 統計方法（資料充足時）

```r
customer_dynamics = case_when(
  ni == 1 ~ "newbie",                              # 新客（只買過 1 次）
  z_i >= 0.5 & r_value <= μ_ind ~ "active",       # 主力客（活躍且最近有買）
  z_i >= -1.0 ~ "sleepy",                          # 瞌睡客（活躍度下降）
  z_i >= -1.5 ~ "half_sleepy",                     # 半睡客（快流失）
  TRUE ~ "dormant"                                  # 沉睡客（已流失）
)
```

#### 方法 B：固定閾值方法（資料不足時）

```r
customer_dynamics = case_when(
  ni == 1 ~ "newbie",
  r_value <= 7 ~ "active",        # 7 天內有買
  r_value <= 14 ~ "sleepy",       # 8-14 天
  r_value <= 21 ~ "half_sleepy",  # 15-21 天
  TRUE ~ "dormant"                 # 超過 21 天
)
```

#### 自動選擇方法的條件

| 條件 | 使用方法 |
|------|----------|
| 回購客戶 ≥ 30 且 觀察期 ≥ 365 天 | Z-Score |
| 其他情況 | 固定閾值 |

---

### 4. 價值等級分類 (Step 8.5)

#### 使用 P20/P80 分位數

```r
m_p20 = quantile(m_value, 0.2)  # 第 20 百分位
m_p80 = quantile(m_value, 0.8)  # 第 80 百分位

value_level = case_when(
  m_value >= m_p80 ~ "高",   # 前 20% 高價值客戶
  m_value >= m_p20 ~ "中",   # 中間 60% 客戶
  TRUE ~ "低"                 # 後 20% 低價值客戶
)
```

#### 特殊處理

| 情況 | 處理方式 |
|------|----------|
| 所有客戶金額相同 | 強制均分成三群 |
| P20 = 最小值 | 調整確保有「低」群 |
| P80 = 最大值 | 調整確保有「高」群 |

---

### 5. 基礎 RFM 指標 (Step 5)

同時計算每位客戶的基本指標：

```r
customer_summary:
  - ni: 購買次數
  - r_value: 最近購買天數（距今幾天）
  - m_value: 平均每次購買金額（AOV）
  - f_value: 購買頻率（= ni）
  - ipt: 平均購買間隔
  - customer_age_days: 客戶年齡（首購至今天數）
  - total_spent: 總消費金額
```

## 輸出結果

函數返回一個 **list** 包含：

### 1. customer_data（客戶資料）

每位客戶包含：

| 欄位 | 說明 |
|------|------|
| 基礎指標 | ni, r_value, m_value, f_value |
| 動態指標 | z_i, F_i_w |
| 分類結果 | `customer_dynamics`（英文）、`value_level`（中文）|

### 2. parameters（分析參數）

| 參數 | 說明 |
|------|------|
| μ_ind | 產業中位購買間隔 |
| W | 活躍觀察窗 |
| λ_w | 平均購買次數 |
| σ_w | 購買次數標準差 |
| method | z_score 或 fixed_threshold |

### 3. validation（資料驗證）

| 欄位 | 說明 |
|------|------|
| 總客戶數 | 分析的客戶數量 |
| 新客數 | ni = 1 的客戶數 |
| 回購客數 | ni ≥ 2 的客戶數 |
| is_valid | 是否有足夠資料使用 Z-Score |

## 關鍵設計特點

### 1. 自動降級機制

```
資料不足（< 30 回購客戶或觀察期 < 365 天）
    ↓
自動切換到固定閾值方法
```

### 2. 產業適應性

- 不用硬編碼 7/14/21 天
- 根據產業實際購買間隔（μ_ind）自動調整

### 3. 統計穩健性

- 使用 Z-Score 標準化，避免絕對值誤判
- 處理邊界情況（σ_w = 0、所有金額相同等）

### 4. 語言輸出

| 欄位 | 輸出語言 |
|------|----------|
| customer_dynamics | **英文**（newbie/active/sleepy/half_sleepy/dormant）|
| value_level | **中文**（高/中/低）|

英文便於後續程式邏輯判斷，顯示給用戶時再轉換成中文。

## 實際應用範例

假設某電商資料：

**輸入**：
- 10,000 筆交易
- 5,000 位客戶
- 觀察期 400 天

**分析過程**：

```
1. μ_ind = 35 天（產業中位購買間隔）
2. W = 91 天（13 週活躍觀察窗）
3. λ_w = 2.3 次（平均在 91 天內買 2.3 次）
4. σ_w = 1.5 次（標準差）
```

**分類結果**：

```
newbie: 3,200 位（64%）- 只買過 1 次
active: 800 位（16%）- z_i > 0.5 且最近有買
sleepy: 500 位（10%）- z_i 在 -1.0 到 0.5
half_sleepy: 300 位（6%）- z_i 在 -1.5 到 -1.0
dormant: 200 位（4%）- z_i < -1.5
```

## 重要：m_value 的定義

⚠️ **注意**：在這個函數中，`m_value` 會被**重新定義**為 AOV！

```r
# Line 257
m_value = m_value / ni  # Convert total to AOV
```

所以函數輸出的：
- `m_value` = **平均訂單金額（AOV）**
- `total_spent` = **總消費金額**

這是很重要的設計決策，影響後續所有模組的計算。
