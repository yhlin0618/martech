## D001: ni < 4 客戶處理方式

### 決策內容
交易次數少於4次的客戶如何在活躍度分析中處理？

### 選擇方案
**方案C: 分層顯示**

### 實施細節
```r
# 1. ni >= 4 的客戶：正常顯示於九宮格
customers_sufficient <- filter(customer_data, ni >= 4)
# 使用 CAI 計算活躍度

# 2. ni < 4 的非新客：單獨顯示區塊
customers_insufficient <- filter(customer_data, ni < 4, lifecycle_stage != "newbie")
# 標註「交易次數不足，使用簡化活躍度」
# 使用 r_value 降級策略

# 3. 新客（ni == 1）：在客戶狀態模組顯示
newbie_customers <- filter(customer_data, lifecycle_stage == "newbie")
# 專屬新客分析區塊
```

### 理由
- ✅ 符合PDF要求（顯示「理論上交易次數少於四次無法算活躍度」）
- ✅ 保留數據完整性（不刪除40-60%的客戶）
- ✅ 邏輯清晰（分層處理不同群體）
- ✅ 提供降級策略（ni < 4 仍可參考）

## D002: 價值等級計算方式

### 決策內容
價值等級應該用哪種計算方式？

### 選擇方案
**方案A: 保持M值單獨計算（過去價值）**

### 實施細節
```r
# 當前實作保持不變
value_level <- case_when(
  m_value >= quantile(m_value, 0.8, na.rm = TRUE) ~ "高",
  m_value >= quantile(m_value, 0.2, na.rm = TRUE) ~ "中",
  TRUE ~ "低"
)

# 在文檔和UI中明確說明
# "價值等級：基於歷史總消費金額（過去價值）"
```

### 理由
- ✅ 符合當前「Value × Activity」命名（Value指歷史貢獻）
- ✅ 簡單直觀，易於理解
- ✅ 識別高價值流失客戶（即使R值很高，M值仍能反映重要性）
- ✅ 業務上合理（用於評估客戶的累積貢獻）

### 未來擴展
