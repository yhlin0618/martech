# GAP-001: 新客數量未顯示問題分析

**Bug ID**: GAP-001
**嚴重程度**: 🔴 Critical
**發現日期**: 2025-10-25
**狀態**: ⏳ 待決策
**影響模組**: Module 1 (DNA 九宮格分析)

---

## 🎯 問題摘要

用戶報告「新客應該是交易次數為1的人，但沒有顯示」。經過深入分析，發現這不是程式邏輯錯誤，而是**新客定義過於嚴格導致實際業務資料中很難有客戶符合條件**。

---

## 🔍 技術分析

### 當前新客定義

**Work_Plan Task 4.1 定義**:
```
新客 = 只買一次 AND 在平均購買週期內
```

**程式碼實現** ([module_dna_multi_premium.R:406](../modules/module_dna_multi_premium.R#L406)):
```r
ni == 1 & customer_age_days <= avg_ipt ~ "newbie"
```

其中：
- `ni`: 購買次數（Number of Transactions）
- `customer_age_days`: 客戶年齡（距離首次購買的天數）
- `avg_ipt`: 平均購買週期（使用中位數計算）

### 問題診斷結果

#### 測試資料 1：原始測試資料（sample_customer_data.csv）
```
總客戶數: 997
單次購買客戶數: 31
平均購買週期 (avg_ipt): 234.25 天

單次購買客戶的 customer_age_days 範圍: 313 - 1003 天
最年輕的單次購買客戶: 313 天

結果: 0 位新客（0%）
原因: 所有單次購買客戶都超過平均購買週期（313天 > 234.25天）
```

#### 測試資料 2：真實場景資料（realistic_customer_data.csv）
```
總客戶數: 1000
單次購買客戶數: 193
平均購買週期 (avg_ipt): 21.58 天

單次購買客戶的 customer_age_days 範圍: 30 - 547 天
最年輕的單次購買客戶: 30 天

結果: 0 位新客（0%）
原因: 所有單次購買客戶的首次購買時間都超過平均購買週期（30天 > 21.58天）
```

---

## 📊 根本原因分析

### 為什麼會出現這個問題？

#### 1. 平均購買週期（avg_ipt）的計算方式

**當前計算邏輯**:
```r
# 計算每位客戶的 IPT
ipt_value = customer_age_days / (ni - 1)  # 對於 ni > 1 的客戶

# 計算全體平均 IPT（使用中位數）
avg_ipt = median(ipt_value, na.rm = TRUE)
```

**問題**:
- `avg_ipt` 是基於**多次購買客戶**的平均購買間隔
- 這個值反映的是「已經回購客戶」的行為模式
- 但用來判斷「第一次購買且還沒回購的客戶」是否為新客

**邏輯矛盾**:
- 活躍客戶的 IPT 很短（例如：10-30天）
- 這導致 avg_ipt 很短（例如：20天）
- 但一個真正的新客可能在60天前第一次購買，還在考慮是否回購
- 結果：這個客戶不符合「< 20天」的條件，被判定為沉睡客

#### 2. 業務現實 vs 數學定義

**數學定義邏輯**:
```
IF (只買一次 AND 距離首次購買 <= 多次購買客戶的平均回購間隔)
THEN 新客
ELSE 沉睡客
```

**業務現實**:
- 新客通常需要更長的時間決定是否回購（60-90天）
- 多次購買客戶已經建立購買習慣，IPT 較短（7-30天）
- 用多次購買客戶的 IPT 判斷新客，標準過於嚴格

**範例**:
```
場景：一位客戶在45天前第一次購買
- 業務角度：這是新客，還在觀察期
- 系統判定：avg_ipt = 20天，45天 > 20天 → 沉睡客 ❌
```

## 💡 解決方案選項

### 方案 A：固定時間窗口（推薦⭐⭐⭐⭐⭐）

**邏輯**:
```r
ni == 1 & customer_age_days <= 60 ~ "newbie"
```

**優點**:
- ✅ 簡單清晰，易於理解
- ✅ 符合業務常識（60天內首購 = 新客）
- ✅ 穩定可預測，不受資料分佈影響
- ✅ 可彈性調整（30/60/90天）

**缺點**:
- ❌ 不是「動態」的（但這可能是優點）

**適用場景**: 大多數電商、零售、服務業

---

### 方案 B：放寬倍數（次推薦⭐⭐⭐⭐）

**邏輯**:
```r
ni == 1 & customer_age_days <= (avg_ipt * 2.5) ~ "newbie"
# 或
ni == 1 & customer_age_days <= (avg_ipt * 3) ~ "newbie"
```

**優點**:
- ✅ 保留「動態」特性
- ✅ 考慮新客需要更長決策時間
- ✅ 適應不同業務的資料特性

**缺點**:
- ❌ 倍數選擇需要經驗判斷
- ❌ 仍然依賴 avg_ipt 的穩定性

**測試結果**（假設 avg_ipt = 21.58天）:
- 2.5倍 = 53.95天 → 預期可識別出部分新客
- 3倍 = 64.74天 → 預期可識別出更多新客

**適用場景**: 希望保持動態性但需要更寬鬆標準的業務

---

### 方案 C：混合方案（最靈活⭐⭐⭐⭐⭐）

**邏輯**:
```r
ni == 1 & customer_age_days <= max(avg_ipt * 2.5, 60) ~ "newbie"
```

**優點**:
- ✅ 結合動態性和穩定性
- ✅ 有最低保障（至少60天）
- ✅ 適應不同資料分佈

**缺點**:
- ❌ 稍複雜

**適用場景**: 不確定業務特性，希望兼顧各種情況

---

### 方案 D：雙條件（彈性最大⭐⭐⭐）

**邏輯**:
```r
# 新客條件：只買一次，且符合以下任一條件
# 1. 在平均購買週期的 3 倍內
# 2. 在 90 天內（絕對時間窗口）
ni == 1 & (customer_age_days <= (avg_ipt * 3) | customer_age_days <= 90) ~ "newbie"
```

**優點**:
- ✅ 非常彈性
- ✅ 適應各種edge cases

**缺點**:
- ❌ 複雜度較高
- ❌ 可能產生混淆

---

### 方案 E：維持現狀 + 文檔說明（不推薦⭐）

**邏輯**:
```r
# 保持原樣
ni == 1 & customer_age_days <= avg_ipt ~ "newbie"
```

**做法**:
- 在文檔中說明此定義的嚴格性
- 提醒使用者可能沒有新客（這是正常的）
- 提供說明：「如果沒有新客，表示所有單次購買客戶都已超過平均回購週期」

**優點**:
- ✅ 符合 Work_Plan 原始定義
- ✅ 不需修改程式碼

**缺點**:
- ❌ 實際使用中幾乎沒有新客
- ❌ 不符合業務直覺
- ❌ 用戶體驗差
