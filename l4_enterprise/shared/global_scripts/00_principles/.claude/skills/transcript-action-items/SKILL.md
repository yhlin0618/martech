# 會議逐字稿行動項目提取

## 目標資源

| 類型 | 名稱 | 來源 | 用途 |
|------|------|------|------|
| 行事曆 | ai行銷 | iCloud | 會議、截止日、里程碑 |
| 提醒事項 | ai行銷 | iCloud | 待辦任務、行動項目 |

> **不使用 Things 3**：Things 3 是個人工具，無法與團隊共享。
> 本 skill 的所有行動項目統一寫入 iCal「ai行銷」，
> 包含純規劃性質（無特定日期）的任務。

---

## 流程

### Step 1：讀取校正後的 SRT

1. **確認目標檔案**
   - 預設目錄：`ai_martech/docs/records/`
   - 只處理 `_corrected.srt` 檔案（校正後版本）
   - 如果用戶未指定檔案，列出可用的 `_corrected.srt` 讓用戶選擇

2. **讀取完整逐字稿**
   - 注意：校正後的 SRT 可能很長（300+ 區塊），需要分段讀取
   - 記錄會議日期（從檔名 YYYYMMDD 取得）

### Step 2：提取行動項目

掃描逐字稿，尋找以下模式：

#### 2.1 行事曆事件（有明確日期/時間）

尋找包含以下線索的段落：

| 線索類型 | 範例 |
|---------|------|
| 明確日期 | 「3月2號」「2/12」「下禮拜三」 |
| 明確時間 | 「晚上10:30」「下午兩點」 |
| 會議安排 | 「跟XX開會」「約個時間討論」 |
| 截止日 | 「X號之前要完成」「deadline 是...」 |

**提取資訊**：
- 事件標題（從上下文推斷）
- 日期（相對日期需根據會議日期換算為絕對日期）
- 時間（如有）
- 相關人員
- 來源區塊編號（方便回溯）

#### 2.2 提醒事項（待辦任務/行動項目）

尋找包含以下線索的段落：

| 線索類型 | 範例 |
|---------|------|
| 指派動作 | 「你去做...」「我來處理...」「XX負責...」 |
| 決策項目 | 「先做XX再做YY」「決定改用...」 |
| 待確認 | 「要確認一下...」「查一下...」 |
| 後續追蹤 | 「之後再看...」「下次討論...」 |

**提取資訊**：
- 任務標題（簡潔描述）
- 負責人（如有）
- 截止日（如有）
- 優先級推斷：1=高（有截止日或明確指派）、5=中（決策項目）、9=低（後續追蹤）
- 來源區塊編號

#### 2.3 排除項目

以下不提取：
- 純粹的過去事實描述（「上次我們做了...」）
- 模糊的未來意向（「以後可能會...」）
- 閒聊中的非承諾性提及

### Step 3：分類標籤

#### 3.1 查詢既有分類

列出「ai行銷」提醒事項清單中已使用的 `[分類]` 前綴：

```
mcp__plugin_che-ical-mcp_ical__list_reminders(
  calendar_name="ai行銷", filter="incomplete", sort="title"
)
```

掃描所有 title 中的 `[...]` 前綴，彙整為「已有分類」清單。

#### 3.2 預設分類

若清單中尚無分類，或需要新增，以下為預設分類：

| 前綴 | 用途 | 適用範例 |
|------|------|---------|
| `[開發]` | 功能開發、bug 修復、部署 | TagPilot 開發、儀表板串接 |
| `[研究]` | 長期技術研究、評估 | R Shiny 替代語言、Supabase |
| `[行銷]` | 文宣、推廣、客戶溝通 | 訂閱版文宣、好農推廣 |
| `[基建]` | 基礎建設、工具、環境 | Telegram 機器人、DigitalOcean 遷移 |
| `[模型]` | AI 評分、精準行銷模型 | Poisson 分析、首購/復購模型 |

#### 3.3 自動建議分類

對每個提取的提醒事項，根據內容自動建議一個分類前綴。
建議邏輯：
- 提到程式開發、功能、修復 → `[開發]`
- 提到研究、評估、探索、替代方案 → `[研究]`
- 提到文宣、推廣、客戶、簡報 → `[行銷]`
- 提到工具、環境、伺服器、部署 → `[基建]`
- 提到模型、評分、演算法 → `[模型]`
- 無法判斷 → 不加前綴，在確認時詢問使用者

### Step 4：呈現給使用者確認

將提取結果整理為表格，**包含分類欄**，請使用者確認、修改或刪除：

```
已有分類：[開發] [研究] [行銷] [基建] [模型]
（可沿用、修改、或新增分類）

## 行事曆事件

| # | 標題 | 日期 | 時間 | 備註 | 來源 |
|---|------|------|------|------|------|
| 1 | 與技詮科技開會 | 2026-03-02 | (未定) | | 區塊 #150 |
| 2 | 線上討論 | 2026-02-26 | 22:30 | 林郁翔+che | 區塊 #380 |

## 提醒事項

| # | 分類 | 標題 | 負責人 | 截止日 | 優先級 | 來源 |
|---|------|------|--------|--------|--------|------|
| 1 | [開發] | 先做技詮版 TagPilot + VitalSigns | che | 3/2 前 | 高 | #200 |
| 2 | [基建] | 停掉 DigitalOcean，改用 Supabase | che | - | 中 | #350 |
| 3 | [行銷] | 好農推廣文宣 | 林郁翔 | - | 低 | #120 |

請確認以上項目，或告訴我需要修改/刪除哪些。
也可以修改分類，或新增分類（如 [客戶]、[合約] 等）。
確認後我會加入「ai行銷」行事曆和提醒事項。
```

**確認迴圈**：
- 使用者說「OK」「確認」→ 進入 Step 5
- 使用者修改某項 → 更新後重新呈現
- 使用者刪除某項 → 移除後重新呈現
- 使用者新增項目 → 加入後重新呈現
- 使用者修改分類 → 更新分類後重新呈現
- 使用者新增分類 → 加入預設清單並分配

### Step 5：寫入 iCal

#### 5.1 行事曆事件

使用 `mcp__plugin_che-ical-mcp_ical__create_events_batch` 批次建立：

- `calendar_name`: `"ai行銷"`
- `calendar_source`: `"iCloud"`
- `start_time` / `end_time`: ISO 8601 格式，時區 `+08:00`
- 若只知道日期不知時間：設為全天事件（`all_day: true`）
- 若有時間但無結束時間：預設 1 小時
- `notes`: 包含來源資訊（如「來自 20260221 週會逐字稿 #150」）

#### 5.2 提醒事項

使用 `mcp__plugin_che-ical-mcp_ical__create_reminders_batch` 批次建立：

- `calendar_name`: `"ai行銷"`
- `calendar_source`: `"iCloud"`
- `title`: **`[分類] 任務標題`** 格式（分類前綴 + 空格 + 標題）
- `due_date`: 若有截止日則設定，否則省略
- `priority`: 根據 Step 2 推斷的優先級
- `notes`: 包含負責人和來源資訊

#### 5.3 回報結果

建立完成後，回報成功/失敗狀態：

```
已成功加入：
- 行事曆：2 個事件
- 提醒事項：3 個任務（[開發] x1, [基建] x1, [行銷] x1）

失敗項目：（如有）
```

## 日期換算規則

會議中常使用相對日期，需根據會議日期換算：

| 口語 | 換算方式 |
|------|---------|
| 「明天」 | 會議日期 + 1 天 |
| 「下禮拜X」 | 會議日期後的下一個星期X |
| 「這禮拜X」 | 會議日期所在週的星期X |
| 「X月X號」 | 直接使用，年份取會議年份（若已過則取下一年）|
| 「月底」 | 該月最後一天 |
| 「下個月」 | 不精確，標記為提醒事項而非行事曆事件 |

**重要**：若日期不確定，寧可設為提醒事項（無截止日）而非猜測行事曆日期。

## 注意事項

1. **寧少勿多**：只提取明確的行動項目，不要過度解讀
2. **保留來源**：每個項目都標註 SRT 區塊編號，方便使用者回溯
3. **尊重使用者判斷**：提取結果只是建議，使用者確認後才執行
4. **相對日期必須換算**：「下禮拜三」要換算成實際日期
5. **時區一律 +08:00**（台灣時間）
6. **批次操作優先**：使用 `_batch` 版本的 API 以提高效率
7. **分類一致性**：優先使用既有分類，避免同義不同名（如 `[開發]` vs `[dev]`）
8. **不使用 Things 3**：所有項目統一寫入 iCal「ai行銷」
